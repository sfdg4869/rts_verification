# RTS 조회/제어 계정 분리 설계

## 1. 배경

현재 구조는 조회와 제어가 충분히 분리되지 않아, 화면에 보이는 계정 입력 UI와 실제 백엔드 동작이 일치하지 않는다.

지금 구현은 대체로 다음과 같다.

- 조회 계열 기능은 공통 SSH 계정으로 수행된다.
- MaxGauge 계열 프로세스 제어도 사실상 공통 SSH 계정 한 쌍으로 처리된다.
- Oracle 관련 작업은 별도 Target DB 계정 흐름이 일부 존재하지만, 프로세스 제어 정책으로 일관되게 강제되지는 않는다.

이 때문에 화면에 프로세스별 ID/PW 입력창이 있더라도, 실제로는 정책이 백엔드에서 제대로 반영되지 않을 수 있다.

## 2. 목표

조회와 제어를 명확히 분리한다.

- 조회(Check/Status)는 공통 계정으로 수행한다.
- 시작/정지(Start/Stop)는 프로세스별 제어 계정으로 수행한다.
- 제어 계정이 없으면 fallback 없이 명시적으로 실패시킨다.
- UI에서 입력받는 계정과 백엔드에서 실제 사용하는 계정을 1:1로 맞춘다.

## 3. 정책

### 3.1 조회 계정

조회 계정은 공통 1세트만 사용한다.

대상:
- RTS 상태 조회
- CPU/MEM 조회
- 기타 read-only SSH 점검
- 상태 확인 목적의 Oracle/프로세스 조회

특징:
- 시작/정지에는 사용하지 않는다.
- 조회 성공 여부와 제어 권한은 별개로 본다.

### 3.2 제어 계정

제어 계정은 프로세스별로 분리한다.

최소 대상:
- `oracle_db`
- `rts`
- `sndf`
- `obsd`
- `updater`

특징:
- 각 프로세스 start/stop은 해당 프로세스의 제어 계정으로만 수행한다.
- 다른 프로세스 계정이나 조회 계정으로 대체 실행하지 않는다.
- 계정 누락 시 명시적으로 실패한다.

## 4. UI 방향

### 4.1 조회 영역

공통 조회 계정 입력만 유지한다.

예시:
- 조회용 SSH User
- 조회용 SSH Password
- 조회용 SSH Port

### 4.2 제어 영역

프로세스 카드별 제어 계정 입력을 둔다.

예시:
- Oracle DB 제어 계정
- RTS 제어 계정
- SNDF 제어 계정
- OBSD 제어 계정
- Updater 제어 계정

원칙:
- 각 카드의 ID/PW 입력은 실제 해당 프로세스 제어 API payload와 직접 연결되어야 한다.
- UI만 존재하고 백엔드에서 무시되는 구조는 제거한다.

## 5. 백엔드 방향

### 5.1 조회 API

기존 조회 API는 공통 조회 계정만 사용한다.

예:
- 상태 조회
- 점검 실행
- CPU/MEM 조회

### 5.2 제어 API

시작/정지 API는 조회 API와 분리한다.

권장 요청 구조:
- `action`: `start` | `stop`
- `target`: `oracle_db` | `rts` | `sndf` | `obsd` | `updater`
- `control_credentials`:
  - `oracle_db`
  - `rts`
  - `sndf`
  - `obsd`
  - `updater`

각 항목은 다음 구조를 가진다.
- `user`
- `password`
- `port` (필요 시)

### 5.3 검증 규칙

백엔드에서 다음을 강제한다.

- 조회 단계와 제어 단계의 입력 구조 분리
- 대상 프로세스의 제어 계정 누락 시 실행 거부
- 잘못된 프로세스 키 요청 시 실행 거부
- 오류 메시지에 어떤 프로세스의 어떤 계정이 필요한지 명시

## 6. 기대 효과

- 운영 정책이 코드에 명확히 반영된다.
- 조회 권한과 제어 권한이 분리되어 안전하다.
- 화면의 입력 의미와 실제 동작이 일치한다.
- Oracle과 MaxGauge 계열의 제어 정책을 혼동하지 않게 된다.
- 나중에 특정 프로세스만 별도 운영 계정으로 바꿔도 확장 가능하다.

## 7. 구현 범위

이번 변경에서 포함할 내용:
- 조회 계정 / 제어 계정 UI 분리
- 프로세스별 제어 credential payload 도입
- start/stop 제어 API 분리 또는 명확화
- 백엔드 제어 계정 검증 강제
- 기존 조회 기능 read-only 유지

이번 변경에서 제외:
- 제어 계정 자동 fallback
- 제어 계정 자동 추론
- 조회 입력값을 제어 단계에 자동 복사하는 우회 로직

## 8. 테스트 항목

- 조회 계정만으로 기존 조회 기능이 정상 동작하는지 확인
- 각 프로세스 start/stop이 자기 제어 계정으로만 동작하는지 확인
- Oracle DB 제어를 RTS 계정으로 시도했을 때 거부되는지 확인
- 특정 프로세스 제어 계정이 비어 있으면 해당 제어만 실패하는지 확인
- 조회 계정만 입력된 상태에서 제어 시 fallback 없이 실패하는지 확인
- 기존 RTS 점검 기능이 read-only 동작으로 유지되는지 확인

## 9. 결정 사항

확정된 정책:
- 조회는 공통 계정 유지
- 제어는 프로세스별 완전 분리
- 제어 계정 누락 시 명시 계정 필수
- fallback 없음

## 10. 구현 메모

- `Target 점검 실행`은 더 이상 `verify_stop`를 사용하지 않는다.
- Oracle DB 제어는 SSH로 `oracle` OS 계정에 접속한 뒤 `sqlplus / as sysdba`를 호출하는 가정을 둔다.
- MaxGauge 계열 제어는 `rtsctl`을 통해 수행한다.
