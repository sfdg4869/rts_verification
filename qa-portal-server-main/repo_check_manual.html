맥스게이지 리포지토리 수집 검증
Repo 점검 항목_신규 사용 매뉴얼

목적: 이 매뉴얼은 Repo 점검 항목_신규 기능을 처음 사용하는 사람도 단계별로 따라할 수 있도록 작성되었습니다.
이 기능은 Target Oracle DB에서 실제 발생한 성능 데이터가 MaxGauge 수집기를 거쳐 손실 없이 Repo DB에 저장되었는지 자동으로 교차 검증합니다.
목차
1. 화면 접속 및 구성 2. Repo DB 연결 (Step 1) 3. Target DB 선택 (Step 2) 4. SSH 접속 정보 입력 (Step 3) 5. 두번째 Repo 설정 (Step 4) — POL_RTS 선택사항 6. 두번째 Target 선택 (Step 5) — POL_RTS 선택사항 7. 검증 실행 8. 결과 테이블 해석

1. 화면 접속 및 구성

브라우저에서 QA Portal URL에 접속한 뒤 상단 메뉴에서 RTS Check 페이지로 이동합니다.

QA Portal RTS Check DB 설정
① Repo DB 연결 → ② Target DB 선택 → ③ SSH 정보 입력 → ④ 두번째 Repo 설정 (선택) → ⑤ 두번째 Target 선택 (선택) → ⑥ Target 비밀번호 → ⑦ 검증 실행
▲ 화면 좌측에서 위 순서대로 설정 후 실행합니다.

4. SSH 접속 정보 입력 (Step 3)

Target DB 서버에 SSH로 접속하기 위한 정보를 입력합니다.

② Target 점검 실행(RTS 7단계) 시 SSH로 서버에 직접 접속하여 rtsctl 상태, 로그, PID 등을 확인합니다.

5. 두번째 Repo 설정 (Step 4) — POL_RTS 선택 사항

VSQL 수집 프로세스(기본 Repo)와 별도로 운영 중인 POL(Polling) 수집 프로세스가 있다면, 두번째 Repo를 설정하면 결과 테이블에 POL_RTS 컬럼이 채워집니다.
사용하지 않는 경우 이 단계는 건너뜁니다. (POL_RTS 열은 모두 -로 표시됩니다.)

언제 필요한가?
MaxGauge 수집 방식이 VSQL(실시간)과 POL(폴링) 두 가지로 나뉘어 각각 다른 Repo DB에 저장되는 환경에서 사용합니다. 두 Repo의 수집 데이터를 나란히 비교할 때 설정합니다.
4 두번째 Repo 설정 선택 사항 — POL_RTS 컬럼에 사용됩니다
미선택    POL Repo (PG)를 선택하면 신규 Repo 점검 실행 시 POL_RTS 값을 별도 조회합니다.

DB Type에 따라 다음 Step 5의 입력 항목이 달라집니다:
• Oracle Repo 선택 시 → DB ID 입력란이 나타납니다
• PostgreSQL Repo 선택 시 → schema_name 입력란이 나타납니다

7. 검증 실행

모든 정보 입력이 완료되면 Repo 점검 항목_신규 버튼을 클릭합니다.

5 검증 실행

※ Repo 점검 항목_신규는 위 두 기능을 통합하여 Target DB 부하 발생부터 Repo DB 수집 확인까지 자동으로 수행합니다.

실행 진행 단계 (6단계)

버튼 클릭 후 진행 상태가 아래와 같이 표시됩니다. 완료까지 수 분이 소요될 수 있습니다.

진행 상태
✔ Step 1
권한 확인 (ALTER SYSTEM / DBMS_UTILITY / DBMS_SHARED_POOL)
✔ Step 2
테스트 프로시저 생성 (qs_sql_test_proc1~5)
✔ Step 3
PL/SQL LOOP 실행 (각 프로시저 100회)
✔ Step 4
Target DB v$sql 조회
⟳ Step 5
Repo DB 수집 결과 조회 (최대 10분 대기)
Step 6
정리 (프로시저 DROP + Shared Pool PURGE)
진행 중... Step 5 / 6 (Repo DB 조회 대기 중)
Step 5는 최대 10분이 소요될 수 있습니다.
MaxGauge가 ORA_SQL_STAT_10MIN 테이블에 10분 주기로 데이터를 집계하기 때문입니다. 데이터가 없으면 30초 간격으로 최대 20회 재시도합니다.
Step 3에서 실행되는 PL/SQL 내용:
5개 테스트 프로시저를 각 100회씩 실행하며, 각 실행마다 ALTER SYSTEM CHECKPOINT를 수행합니다.
각 프로시저는 지정된 시간(3초 / 1초 / 0.05초 / 0.01초 / 0.001초)만큼 CPU를 점유합니다.
DECLARE TYPE test_func_list IS TABLE OF VARCHAR(30); v_funcs test_func_list := test_func_list( 'qs_sql_test_proc1', 'qs_sql_test_proc2', 'qs_sql_test_proc3', 'qs_sql_test_proc4', 'qs_sql_test_proc5' ); test_execution NUMBER := 100; BEGIN FOR i IN 1..v_funcs.COUNT LOOP FOR j IN 1..test_execution LOOP EXECUTE IMMEDIATE 'ALTER SYSTEM CHECKPOINT'; EXECUTE IMMEDIATE 'BEGIN ' || v_funcs(i) || '; END;'; END LOOP; END LOOP; END;

8. 결과 테이블 해석

검증이 완료되면 아래와 같은 결과 테이블이 출력됩니다. 가로로 3개 영역으로 나뉘며 각 Case별로 3개 행(Row)이 한 묶음입니다.

화면 구성: 결과 테이블은 좌측 패널(Target DB V$SQL 기대값)과 중앙·우측 패널(Repo DB ORA_SQL_ELAPSE / ORA_SQL_STAT_10MIN)로 구성됩니다. 각 Repo는 VSQL_RTS(기본 Repo)와 POL_RTS(두 번째 Repo, 미설정 시 "-") 두 열로 표시됩니다.
