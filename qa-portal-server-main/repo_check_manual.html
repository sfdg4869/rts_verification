2. 버튼별 점검 결과 해석 가이드

5단계에서 어떤 버튼을 눌렀느냐에 따라 생성되는 결과 화면과 해석 방법이 다릅니다.

2.1 Target 점검 실행 은 무엇을 점검하나요?

Target DB 서버에 SSH로 직접 접속하여 RTS 수집 에이전트가 정상 작동하는지 7단계로 점검합니다.

  • 1. 로그 디렉터리: rts_log, ocr_log 등 필수 수집 로그 폴더가 존재하는지 검사합니다.
  • 2. rtsctl stat: 에이전트 제어 명령어(rtsctl stat)가 정상 호출되고 상태가 Active인지 확인합니다.
  • 3. PID 교차검증: 로그에 기록된 데몬의 PID 번호가 Target 서버 OS 상에 실제 프로세스로 떠있는지 비교합니다.
  • 4. ERROR grep: 에이전트 동작 로그(*.log) 내부에 치명적인 ERROR 문구가 발생했는지 문자열을 스캔합니다.
  • 5. CPU/MEM: 해당 수집 데몬 프로세스가 OS의 CPU와 메모리를 비정상적으로 많이 점유하지 않는지 확인합니다.
  • 6. 비정상 종료 시그널: 수집기가 과거 비정상적인 시그널(Kill 등)을 받고 종료된 이력이 있는지 점검합니다.
  • 7. Target 조회(v$sql): Target 서버를 향해 10초짜리 미니 쿼리 5번을 쏜 뒤, Target 오라클 내부 `V$SQL` 테이블에 쿼리 소요시간이 정확히 50초로 기록되었는지 DB 단에서 1차 확인합니다.

2.2 Repo 점검 실행 은 무엇을 점검하나요?

Target 수집 에이전트가 쏘아올린 데이터가 매니저 서버의 Repo DB (저장소)에 유실 없이 잘 적재되었는지 6단계로 점검합니다. (이때 오늘 날짜의 파티션 데이터를 스캔합니다.)

  • 1. 누락 재현 여부: APM_TOP_OS_PROCESS_LIST 테이블에 수집기 프로세스 정보가 누락 없이 기록되는지 쿼리합니다.
  • 2. ELAPSE 1초 초과 로깅: ORA_SQL_ELAPSE 테이블에 소요 시간이 1초를 초과하는 무거운 쿼리들이 정상 이관되고 있는지 검증합니다.
  • 3. STAT 50초 이상 로깅: ORA_SQL_STAT_10MIN 모음 테이블에 총 50초 이상 누적된 지표 기록이 존재하는지 스캔합니다.
  • 4. ELAPSED/EXECUTION: 단순히 시간만 찍히는 것이 아니라 쿼리 수행 횟수(Execution)와 소요 시간의 비율이 맞는지 검사합니다.
  • 5. 대상 SQL 수집 확인 (Elapse): Target 검사 7단계에서 쐈던 10초짜리 미니 쿼리 5회의 내역이 개별 실행 단위(Elapse)로 Repo에 무사히 도착했는지 확인합니다.
  • 6. 대상 SQL 수집 확인 (Stat): 미니 쿼리 5번의 내역이 10분 단위 통계 테이블(Stat_10Min)에 누적 50초, 5회로 잘 합산 계산되어 적재되었는지 검사합니다.

2.3 Repo 점검 항목_신규 는 무엇을 점검하나요?

위 두 버튼(Target/Repo 점검)을 원스톱으로 통합하여 양쪽 데이터를 1:1로 교차 비교하는 신규 궁극기 버튼입니다.

  • - 부하 주입: Target DB에 11가지의 각기 다른 케이스(CPU Burn, Sleep, Fetch 등)의 악성 쿼리를 강제 실행하여 부하를 줍니다.
  • - 대조 측정: Target DB 내부 시스템 테이블(`V$SQL`)에 찍힌 실제 부하 소요시간을 파이썬이 즉시 캡처합니다.
  • - 수집 검증: 이후 Repo DB의 저장 테이블(`ORA_SQL_ELAPSE`, `STAT`)을 뒤져 동일한 SQL_ID 데이터가 전송되었는지 추출합니다.
  • - 시각화: 양쪽 DB의 지표 수치가 소수점 아래까지 정확히 일치하는지 한눈에 비교할 수 있도록 아래와 같은 대형 병합 표(Grid)를 생성합니다.

📌 표 보는 방법 (핵심)

실제 Target DB(V$SQL)의 지표 값과 수집 서버 내 데이터(VSQL_RTS, POL_RTS)를 나란히 배치하여 세로 3줄(Row 1,2,3) 단위로 1:1 비교 판단을 내릴 수 있습니다.

💡 수집 불량(오류) 판별 방법

1. 수집 누락 현상 검증: 가장 윗줄(Row 1)을 확인했을 때 Target DB의 Executions 값이 10,000 인데, 우측 Repo DB(STAT_10MIN)의 Execution Count 값이 그에 한참 미치지 못하거나 '-' 인 경우 정상 수집이 안 된 것입니다.
2. 시간 오차 검증: 가장 아랫줄(Row 3)에서 Target DB의 평균 타임과 수집서버의 1회당 평균 타임에 심각한 간극이 벌어지면 시간 변환 로직 오류를 의심해야 합니다.
